# ESP32-C3 VTX Controller (SmartAudio / TRAMP / MAVLink Bridge)

This project uses the official MAVLink C library for MAVLink frame parsing and message encoding.

Overview
- ESP32-C3 firmware for controlling one or more VTX devices from PWM inputs.
- Each configured device has its own PWM input pin, VTX control pin, protocol, name, and enabled state.
- The board starts a Wi-Fi access point and serves a web UI for device configuration, VTX table selection, JSON upload, and JSON editing.
- One firmware image supports three board roles: standalone, `fc_bridge`, and `vtx_node`.
- VTX tables are stored as JSON on LittleFS and the currently selected table is also kept in memory for fast access from the UI.

How It Works
- Each enabled device reads its own PWM input and maps the pulse width to a slot in the current VTX table.
- Commands are sent to multiple VTX devices in a serialized manner. The ESP32-C3 can manage several VTX devices, but it does not drive all protocols in true parallel at the same instant.
- SmartAudio is sent over a selectable one-wire pin. TRAMP is sent over `Serial1`, retargeted to the selected device control pin before each command.
- Devices can also run in `mavlink` control mode. In that mode they wait for a MAVLink-derived command addressed by `nodeId` and `deviceId`.
- In `fc_bridge` role, the ESP32 listens to MAVLink on a configurable UART, applies commands to any matching local MAVLink devices, and forwards targeted commands to remote ESP-NOW nodes.
- In `vtx_node` role, the ESP32 ignores FC UART and only applies `mavlink_apply` commands received through ESP-NOW.

Web UI
- Join the access point `ESP32C3-VTX` with password `13579135`.
- Open the AP IP shown on serial output, typically `http://192.168.4.1/`.
- From the UI you can:
  - choose the board role and local MAVLink node ID
  - configure MAVLink UART RX/TX pins and baud rate for the FC bridge role
  - add, edit, enable, disable, and remove VTX device entries
  - choose PWM input pin, VTX control pin, protocol, and control mode per device; set PWM to -1 to disable PWM capture for that device
  - assign per-device MAVLink `nodeId` and `deviceId`
  - select the active VTX table
  - upload a new VTX table JSON file
  - load and edit the selected VTX table JSON directly in the browser

MAVLink Command
- The firmware listens for `COMMAND_LONG` with command ID `31001`.
- Parameter layout:
  - `param1`: target `nodeId` (`255` means broadcast)
  - `param2`: target `deviceId` (`255` means all devices on the node)
  - `param3`: band (`0` keeps current configured band)
  - `param4`: channel (`0` keeps current configured channel)
  - `param5`: power index (`255` keeps current configured power index)
  - `param6`: flags (currently reserved)
  - `param7`: reserved
- The FC bridge always applies matching local MAVLink devices first. If the target node is not the local node, or is broadcast, the command is also forwarded over ESP-NOW.

Hardware
- Connect one PWM source per enabled device to its configured PWM input pin.
- Set a device PWM input pin to -1 when the device should stay enabled but not consume a PWM input.
- Connect each VTX control pin to the matching VTX SmartAudio/TRAMP control line.
- Do not assign the same PWM input pin or the same VTX control pin to multiple enabled devices.
- For `fc_bridge` role, connect the configured MAVLink RX pin to the flight controller TX pin. The MAVLink TX pin is optional and may be set to `-1` if unused.
- Many VTXs require specific wiring, voltage levels, inversion, or half-duplex handling. Check the VTX documentation before connecting hardware.

Build
- Use PlatformIO from the project root.

```bash
pio run
pio run -t upload
pio device monitor
```

MAVLink Serial Test
- A dedicated UART test sketch is available in `src/mavlink_serial_test.cpp`.
- Build it with the `esp32-c3-devkitm-1-mavlink-test` environment.
- The sketch uses UART1 on GPIO5 RX and GPIO6 TX at `115200` baud.
- Wire the flight controller TX to ESP32 GPIO5 and the flight controller RX to ESP32 GPIO6.
- The test sketch identifies itself as MAVLink system `2`, component `158`, sends heartbeats once per second, logs incoming frames over USB serial, and returns `COMMAND_ACK` accepted for custom command `31001`.

Notes
- Configuration is stored in `/config.json` on LittleFS.
- VTX tables are stored as `.json` files on LittleFS.
- SmartAudio and TRAMP timing and framing may still need adjustment for specific VTX models.
- ESP-NOW forwarding uses broadcast transport with node/device filtering in the payload, so no explicit peer MAC configuration is required in this version.
